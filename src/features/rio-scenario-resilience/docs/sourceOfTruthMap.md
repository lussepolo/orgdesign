# Rio Scenario Resilience Source-Of-Truth Map

This document maps the Scenario Resilience scaffold to the existing Rio app model.
It is metadata only. It does not add calculations, assumptions, or new formulas.

## 1) Summary

The current app already contains reusable payroll, staffing, tuition, and viability logic.
The scaffold can reuse those sources for payroll, staffing, role-cost, FOPAG / Folha Direta, OPEX, CAPEX, VPL/NPV, payback, and sensitivity interpretation.
Receita is not a fixed existing app output. It must be generated by the simulator from the selected lever combination.
What is still missing is a dedicated EBITDA layer, a confirmed discounted-payback layer, and any tiering rules.

## 3) What Is Missing

- EBITDA calculation logic
- EBITDA margin calculation logic
- Confirmed discounted-payback logic
- Tiering rules and scoring
- A dedicated service-contract source of truth
- A lever-level CAPEX package, if the scaffold needs one later

## 4) Revenue Correction

- Receita is not a fixed input from existing payroll.
- Existing payroll should only be used for cost-side logic.
- The simulator's revenue output will be generated from the chosen combination of opening grades, occupancy, tuition, discount assumptions, and annual tuition adjustment assumptions.
- This prevents old revenue assumptions from overriding the resilience simulator's lever-based design.

## 5) What Must Not Be Duplicated

- Payroll formulas already in `src/lib/payroll/core.ts`
- Cost-side payroll formulas already in `src/lib/payroll/domain.ts`
- OPEX and CAPEX formulas already in `src/lib/viability/baseline.ts`
- Staffing formulas already in `src/hooks/useStaffingLogic.ts`
- Viability state and baseline assumptions already in `src/hooks/useViabilitySimulator.ts`
- Any new placeholder salary, tuition, CAPEX, or service-contract assumptions

## 6) Integration Risks

- Opening Grades are confirmed in the scaffold, but they still need to be mapped onto the live grade schedule.
- Occupancy already exists in the app model, but it is not exposed as a standalone lever input.
- Org Design Structure is defined as blocked option shells because Scenario Offer role activation data is narrative-only. It must remain blocked from financial outputs until machine-readable role activation records and role-cost assumptions are validated.
- CAPEX already exists in the viability model, but it is an aggregate assumption rather than a lever-level structure.
- Discounted Payback is not clearly implemented, so it should remain unresolved until confirmed.
- Scenario Tier remains a future interpretation layer only.

## 7) Recommended Next Integration Sequence

1. Confirm lever data inputs
2. Map existing payroll and revenue logic
3. Map existing viability baseline
4. Define missing EBITDA layer
5. Define discounted payback only if required
6. Define scenario tier interpretation rules
7. Only then wire a calculation layer
